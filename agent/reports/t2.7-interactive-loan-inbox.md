# Plan — T2.6 full-spec upgrade (interactive missed-loan inbox)

## Context

Phase 2 Track A's T2.6 landed on `t2_track_a` in an MVP shape: when a
`loan_due` day arrives and the player can't pay, an inbox entry is pushed
with `options: []` — purely informational, no player agency. The spec in
`agent/TASKS.md` calls for "body + options built from a small template"
that the player can act on from the Email app.

This plan replaces the informational-only inbox entry with an interactive
one carrying four real options: **pay now**, **pay from savings**, **take
a payday loan to cover**, **let it slide**. The `/api/event/resolve`
endpoint grows a small router so synthetic `cal_*` events use a dedicated
resolver (non-SAGE-shaped — no d20 roll, real side-effects like creating a
payday loan).

Starting point: branch off `t2_track_a` (or `main` once that branch is
merged). New branch: `t2_6_interactive`.

## Design

### Synthetic inbox entry shape

In `events._fire_loan_due`, when `make_loan_payment` returns MISSED, build
a richer event payload and push via `sage.push_to_inbox` (existing helper).

Shape reuses the SAGE option schema so the UI template at
`templates/index.html:330` iterates it unchanged:

```python
{
    "slug": f"cal_loan_due_m{ev.month}",
    "sender": "Bank",
    "title": f"Missed {loan.kind} loan payment — action required",
    "body": "... 150 PLN shortfall. Pick a way out.",
    "options": [
        {"id": "opt_pay_now",      "label": "Pay from checking",
         "skill_check": None, "hint": "Only works if you have the cash.",
         "effects_on_success": {"money": -loan.monthly_payment},
         "effects_on_failure": {"money": -loan.monthly_payment}},
        {"id": "opt_from_savings", "label": "Transfer from savings",
         "skill_check": None, "hint": "Raid the emergency fund.",
         "effects_on_success": {"money": 0}, "effects_on_failure": {"money": 0}},
        {"id": "opt_payday_loan",  "label": "Take a payday loan to cover",
         "skill_check": None, "hint": "120% APR. Genuinely bad idea.",
         "effects_on_success": {}, "effects_on_failure": {}},
        {"id": "opt_skip",         "label": "Let it slide",
         "skill_check": None, "hint": "Credit score takes a hit.",
         "effects_on_success": {"credit_score": -15, "sanity": -5},
         "effects_on_failure": {"credit_score": -15, "sanity": -5}},
    ],
}
```

The effect dicts on options are **display-only previews** so
`successPct()` / "100% success" pill / effect-preview rendering in Track C
stays consistent. Authoritative effects are computed in the resolver.

`event_id` stays `cal_loan_due_{month}_{day}_{loan_index}` so the resolver
can parse the loan index back out. Persistence: current code drops the
calendar entry after firing — keep that. The inbox entry itself is the
persistent piece until the player clicks an option.

Dedup: if `_fire_loan_due` is somehow called twice for the same
`(month, day, loan_index)` in one session (shouldn't happen with
calendar-drop, but defensive), check `state.inbox` for an existing
unresolved entry with the same `event_id` and skip.

### Resolver routing

`/api/event/resolve` in `api.py` inspects `event_id`. If it starts with
`cal_`, dispatch to `events.resolve_calendar_event(state, event_id,
option_id)`. Otherwise fall through to the existing
`sage.resolve_event(...)` path.

The client always rolls and sends `roll_d20`; for `cal_*` events the
server ignores it. The UI's resolution panel (`index.html:354–388`) shows
"no check" branch when `resolution.skill` is null — matches today's
behavior.

### `events.resolve_calendar_event`

```python
def resolve_calendar_event(state, event_id, option_id) -> dict:
    # Parse event_id: "cal_loan_due_{month}_{day}_{loan_index}"
    # Find inbox ref, validate not resolved.
    # Find loan by index; if already paid (remaining <= 0), mark resolved
    #   with an informational payload and return.
    # Dispatch on option_id:
    #   opt_pay_now      -> finance.make_loan_payment(state, i)
    #   opt_from_savings -> move max(0, pay - checking) from savings, then pay
    #   opt_payday_loan  -> finance.take_loan(state, "payday", pay),
    #                       then finance.make_loan_payment(state, i)
    #   opt_skip         -> sage.apply_effects(state, {credit_score:-15, sanity:-5})
    # Mark inbox entry resolved. Return {passed, effects_applied, note}.
```

Returns a shape compatible with `resolveOption` in `quid.js:367` — the
resolution panel only reads `passed`, `effects_applied`, `skill`, `dc`,
`total`. Setting `skill=None, dc=None, total=None, passed=True` renders
the "no check" variant cleanly.

**Available options:** all four are always shown. If the player picks
`opt_pay_now` and checking is still insufficient, the resolver calls
`make_loan_payment` which flags another miss — resolution reports
`passed=False, note="still couldn't pay"` and the inbox stays unresolved
(user can pick a different option). For `opt_from_savings`, if savings
is also too low, resolver tops up what it can, leaving the remainder
unpaid and reporting partial failure. For `opt_payday_loan`, always
succeeds (crazy APR is the cost). For `opt_skip`, always succeeds —
credit hit applied, inbox resolved.

### Prevent re-fire on re-render

No calendar-based re-fire — calendar entry is dropped at fire time
(current behavior, keep). The inbox entry persists across day ticks
until resolved. Subsequent `loan_due` calendar events for future months
are seeded normally in `_rollover_month`.

### Reuse map (don't duplicate)

- `finance.make_loan_payment(state, i)` — `finance.py:143`
- `finance.take_loan(state, "payday", amount)` — `finance.py:159`
- `sage.apply_effects(state, effects)` — `sage.py:105`
- `sage.push_to_inbox(state, event)` — `sage.py:78`
- `EventRef`, `CalendarEvent` — `state.py`
- UI inbox + options render — `templates/index.html:295,330` (unchanged)
- UI resolution panel — `templates/index.html:354` (unchanged)
- `quid.js:resolveOption` — unchanged; already handles null `skill_check`

## Verification

1. **Miss creates actionable inbox entry.** Seed state with a bnpl loan,
   checking below `monthly_payment`, savings above it; advance to
   `due_day`. Inbox should contain one entry with 4 options, calendar
   entry gone, `loan.payments_missed == 1` (from the auto-attempt).
2. **`opt_from_savings` pays.** Resolve with `opt_from_savings`. Expect:
   savings decreased by shortfall, checking adjusted, loan.remaining
   decreased by `monthly_payment`, inbox marked resolved.
3. **`opt_payday_loan` creates a new loan.** Resolve with
   `opt_payday_loan` starting from broke state. Expect: new Loan with
   `kind="payday"` in `state.loans`, new `loan_due` CalendarEvent seeded
   for it, original loan paid, inbox resolved.
4. **`opt_skip` applies credit penalty.** Resolve with `opt_skip`. Expect:
   credit_score −15, sanity −5 (clamped), inbox resolved, no finance
   calls.
5. **`opt_pay_now` from broke state reports failure.** Resolve with
   `opt_pay_now` while checking is insufficient. Expect: resolution
   `passed=False`, `loan.payments_missed` increments again, inbox stays
   unresolved, player can pick a different option.
6. **UI smoke.** `manage.py runserver` → load a state with the missed
   inbox entry → click each option — resolution panel renders, state
   updates land in localStorage, existing SAGE events still resolve
   through the original path unaffected.
7. **Tests pass.** `python manage.py test game.tests`.
